A cooperative, non-competitive game — players share the same world, rooms are bounded spaces where all occupants can see each other. No hit registration, no per-player visibility filtering, no anti-cheat constraints on message routing. The room boundary is the fan-out boundary. With ~200 players per room, broker egress is orders of magnitude higher than ingest.

Originally HTTP-based. Latency spikes under load, mobile clients dropping connections on any network transition, and fan-out complexity growing with every new player in a room. This talk walks through the architecture that replaced it.

Scope: This architecture fits a specific class of game — cooperative, social, non-competitive. Competitive games with authoritative physics or anti-cheat requirements would position the broker differently (or bypass it for the hot loop entirely).

Layer 1

Regional edge brokers

NA cluster

Players in North America connect to the nearest EMQX edge node. Round-trip from New York to the NA edge: ~8ms. To a central EU cluster: ~90ms.

EU cluster

Same pattern for Europe. Each regional cluster accepts MQTT connections and bridges to the central cluster over an MQTT bridge connection.

APAC cluster

Tokyo, Singapore, Sydney. Mobile-heavy market. Regional edges give mobile players a fast, localized connection point.

Star topology & cross-region rooms

Star topology: Regional edge clusters bridge to a central cluster via stable datacenter-grade MQTT bridge connections — not client-grade links. Inter-cluster bridging is a persistent, high-throughput server-to-server channel with its own reconnect and buffering logic.

Cross-region rooms: When players from different regions join the same room, the authoritative broker for that room is the central cluster. Edge brokers relay messages to/from their local players — adding one regional hop but keeping the fan-out centralized and consistent.

Why not raw WebSockets

The protocol mismatch

WebSockets · SSE

  • A raw pipe — nothing more. WebSocket gives you a bidirectional byte stream. Player location broadcast, interest management (AoI), reliable events vs unreliable updates, reconnect state synchronization: none of that exists in the protocol. It is all your application code.
  • Point-to-point fan-out. 1,000 subscribers means your server calls send() 1,000 times. The protocol has no concept of publish-once-deliver-many.
  • SSE is one-directional — server push only. Client responses require a separate HTTP channel.

MQTT

  • Pub/sub is the protocol. Publishers and subscribers are decoupled by design. The broker handles topic routing, fan-out, and delivery — not your server.
  • Broker fan-out. Publish once; broker delivers to every subscriber. Your game server sends one message regardless of how many clients are in the room.
  • QoS 0 / 1 / 2 — delivery guarantees are a protocol primitive, not application code.
  • Keep-Alive heartbeat — broker detects silent disconnects; mobile radio can sleep between heartbeats.
  • Session persistence — the broker remembers the client's session state across network transitions, avoiding full application-level reconnects if the socket drops.

Caveat: Real production WebSocket architectures use edge proxies (Envoy, HAProxy) that handle TLS, keepalives, and connection pooling — not raw sockets. The more honest comparison is MQTT broker vs. edge-proxy-plus-custom-pub/sub-layer. Slide 7 addresses that trade-off directly.

Operations

Observable by design

HTTP · WebSockets · SSE

  • Standard monitoring still applies. Teams build Grafana/Datadog dashboards regardless of transport. The question is what additional live inspection capability you get from the transport layer itself.
  • Player-specific investigation: filter logs by session_id or player_id. Works, but requires a logging stack (ELK, Splunk, Datadog) and you're still reading text, not watching live traffic.
  • Network-level sniffing (tcpdump, Wireshark) gives full packet visibility but is cumbersome to set up on production hosts and rarely done in practice.
  • A proxy (mitmproxy, sidecar) can intercept traffic in real time but is seldom deployed just for debugging — it is extra infrastructure that sits in the request path permanently.

MQTT — a network X-ray

  • The broker sees the network, not the game. Application-level bugs still need your own tooling. What the broker gives you is visibility into the transport layer: did the client drop the connection, or did the server stop sending? An edge proxy gives this for TCP; the broker gives this per topic.
  • Delivery health is built in. Disconnect Reason Codes, LWT (Last Will), and QoS unacknowledged message queues tell you exactly where the pipeline broke — without touching game server logs or C++ engine code.
  • Live topic inspection for scoped investigation: 
    SUBSCRIBE game/room/42/#
    Watch one room's traffic in real time — no code changes, no redeployment, invisible to players.
  • Built-in packet tracing. The broker natively provides packet tracing, allowing you to easily inspect payload data live in a non-disruptive way. You do not need a separate application or proxy to inspect traffic.
  • Scale caveat: At high message rates, you scope your debug subscription narrowly (game/room/42/player/17/#) or sample. This is targeted investigation, not "subscribe to everything and read it all."
Honest engineering

Trade-offs: latency vs. connection offloading

The cost you pay

  • Latency and Jitter. Client → Broker → Room Server. This adds ~10–20 ms of unloaded latency. However, during flash crowds (e.g., 1,000 players spawning at once), broker topic-matching and fan-out routines can cause unpredictable latency jitter, breaking movement prediction.
  • A heavier edge component. Compared to a lightweight, finely-tuned edge proxy like Envoy or a custom UDP gateway, an Erlang/OTP distributed broker is massive and complex to operate, monitor, and tune.
  • Binary payloads are opaque to the broker. The broker routes bytes; it cannot inspect or validate game-specific serialization. That logic stays in your server.

What you get in return

  • Unified edge layer. Proper game architectures already use edge proxies to handle socket I/O, TLS, and keepalives. The broker replaces those edge pipes with a single layer that handles both transport connectivity and robust pub/sub routing at massive scale.
  • Offloading the fan-out loop. Even with an edge proxy, a typical game server must iterate over memory structures to fan out messages to specific client connections. With a broker, the game loop publishes exactly once per tick. The broker duplicates the payloads to all topic subscribers.
  • Operational leverage. QoS, state synchronization, and data export — each is a feature you'd otherwise build, test, and maintain as custom control-plane logic behind your proxies.
  • Battle-tested infrastructure. EMQX is deployed at scale in IoT, automotive, and telecom — but this workload profile (high-frequency, low-latency game events) is a different beast. What makes it viable here is EMQX's Erlang/OTP foundation: lightweight processes, preemptive scheduling, and soft-realtime GC designed for exactly this class of concurrent messaging.
  • Resilient state recovery. The broker shields the backend from connection chaos and TLS thundering herds, while Retained Messages provide instant secondary state to reconnecting clients without separate REST APIs.
Beyond pub/sub

EMQX Protocol Extensions

MQTT Streams

Persistent, append-only log attached to a topic. Any subscriber can start reading from any offset — 200 clients, 200 independent cursors, one stored history.

SUBSCRIBE $stream/positions/game/{room_id}/#
  [MQTT Property] stream-offset: {last_ts}
# Replay in-order → transition to live feed

Partitioned by key expression (e.g., room_id) with strict per-partition ordering — the core Kafka pattern, inside the broker. For many workloads this replaces a dedicated Kafka cluster. Maintaining Kafka is a non-trivial operational burden, and managed Kafka is expensive. When you do need a full data lake, EMQX bridges natively to Kafka, Snowflake, or ClickHouse.

Message Queue

# Held if player is offline
PUBLISH $queue/player-commands/player/{id}/commands
  {"type":"teleport","dest":"zone_7"}
# Delivered QoS 1 when client reconnects

Offline delivery — messages are persisted even when the target client is disconnected. Commands, notifications, or config updates are held in the queue and delivered reliably (QoS 1) when the client reconnects.

Last-Value semantics — with a queue key expression, only the latest message per key is retained. Ideal for dispatching the most recent configuration, cosmetic loadout, or announcement without stacking stale updates. This is a different mechanism from standard MQTT Retained Messages — Retained Messages deliver one message per topic to new subscribers (latest state); Message Queue Last-Value is per-key dedup in a persistent queue with offline delivery.

Dispatch strategies — distribute queued messages across multiple consumers via round-robin, random, or least-inflight routing. Useful for parallelizing command processing or batch work across a pool of workers.

Discussion

Where else do these patterns apply?

Question

Where in your stack do you have more than a few hundred clients waiting for the same update? Are you polling somewhere today that could be a subscription?

Financial market data

Tick data fan-out to thousands of subscribers from a single publish. Shared subscriptions distribute order-processing across execution engines. Retained Messages gives a late subscriber the current price without a bootstrap REST call.

Collaborative editing

Cursor positions, shape mutations, presence across concurrent editors. The batch-write pattern applies: accumulate cursor-move events, broadcast at display cadence. Not every intermediate cursor position needs to reach every collaborator.

Sports betting / live events

Odds change every few seconds. All subscribers need the latest number simultaneously. One publish to all, sub-second. Retained Messages: a newly opened app sees current odds immediately, no separate price API call.

Messaging & social

"Is typing…", presence, read receipts. For offline delivery, EMQX's Message Queue (not Retained Messages) holds missed messages and delivers them QoS 1 when the client reconnects — no FCM workarounds or separate queue store. Retained Messages gives the latest state per topic (e.g., current status), not full message history.

Summary

Three things to take away

Decouple transport from simulation

  • The broker absorbs 1M+ flaky mobile connections — TLS termination, keepalives, reconnect storms — so the game server never touches raw socket I/O
  • Your game logic processes validated inputs. Fan-out, QoS, session persistence, and shared consumers are the broker's job
  • The trade-off is ~10–20 ms of datacenter latency. For a cooperative social game, that buys enormous operational leverage

Simplifying the backend

  • The broker shields the game server from connection chaos. An edge proxy can hold TCP connections during a backend crash, but the broker holds both the connections and the pub/sub routing state, allowing instant topology recovery.
  • MQTT Streams can replace a dedicated Kafka cluster for many game event workloads — one fewer system to operate and pay for
  • Retained Messages handle secondary state (cosmetics, flags, room weather) on reconnect — the game server doesn't package it

Resilience by design

  • Broker node failure: EMQX's clustered architecture means sessions migrate to surviving nodes. Clients reconnect to another node in the cluster and resume their subscriptions — no application-level orchestration required.
  • Backend crash: The broker holds client connections while game servers restart. Players stay connected; the new server instance subscribes and picks up where the old one left off.
  • Network transitions: Session persistence across IP changes — mobile clients switching between WiFi and cellular resume without full re-authentication or state resync.
