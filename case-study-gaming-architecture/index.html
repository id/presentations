Scope: This fits cooperative/social games. Competitive games with authoritative physics would position the broker differently.

Honest engineering

Trade-offs: latency vs. connection offloading

The cost you pay

  • Latency + jitter. ~10-20 ms unloaded. Flash crowds can cause unpredictable spikes in broker topic-matching
  • Less familiar tooling. Game teams know Redis, Envoy, custom UDP — an MQTT broker is a new component to learn, operate, and debug
  • Opaque payloads. Broker routes bytes — game-specific validation stays in your server

What you get in return

  • Unified edge. One layer for transport connectivity and pub/sub routing — replaces proxy + custom fan-out code
  • Erlang/OTP runtime. Preemptive scheduling, soft-realtime GC, no stop-the-world pauses at 1M connections
  • Session persistence. Mobile clients survive WiFi ↔ cellular without re-auth or full state resync
  • Open standard. MQTT is an OASIS/ISO standard — no vendor lock-in, broad client library support across languages and platforms
  • EMQX Streams + Message Queue. Event replay without Kafka, offline delivery for mobile clients (EMQX-specific extensions — covered in the previous talk)

Summary

Three things to take away

Decouple transport from simulation

  • Broker absorbs 1M+ flaky connections — TLS, keepalives, reconnect storms
  • Game logic processes inputs. Fan-out, QoS, sessions are the broker's job
  • Trade-off: ~10-20 ms latency. Reasonable for cooperative/social games

Simplifying the backend

  • Broker holds connections and routing state during crashes — instant topology recovery
  • EMQX Streams (EMQX extension) can replace Kafka for game event workloads
  • Retained Messages handle secondary state on reconnect — game server doesn't package it

Failure modes and recovery

  • Broker failure: Sessions migrate to surviving nodes, clients resume subscriptions
  • Backend crash: Broker holds connections while servers restart
  • Network switch: Session persistence across WiFi ↔ cellular — no re-auth needed
  • Offline clients: EMQX Message Queue holds missed events, delivers on reconnect
